This document specifies GDIS: Global Digital Identity Scheme as a web-first identity binding profile that starts from a physical, governance-issued identity item and produces verifier-checkable cryptographic evidence.

GDIS does not claim automatic legal recognition in any jurisdiction. It defines how to derive a stable subject key from MRZ-extracted PID hash material, bind that key to governance verification endpoints, issue the binding as a VC to a DID controlled by a GQSCD-class device, and publish verification material in an open decentralized event log.

Scope

This specification covers:

  • identity anchoring to a physical item issued under a governance area, with governance legitimacy modeled as an external fact;
  • deterministic MRZ-extracted PID hash derivation and governance verification endpoint binding;
  • credential binding as a VC issued to a DID where the DID controller is GQSCD-class;
  • open-participation decentralized event-log publication and verifier conflict resolution rules.

This specification does not:

  • create legal recognition by itself;
  • replace jurisdictional trust lists, supervisory registries, or legal status determinations;
  • standardize one specific chain, one specific vendor stack, or one closed host network.

Positioning and Adoption Model

GDIS is a technical interoperability profile. It is designed so that legal recognition claims remain anchored to jurisdictional registries, while cryptographic verification remains mechanically testable by independent verifier implementations.

  • Issuer adoption objective: produce portable binding credentials and evidence artifacts that can be verified without hidden trust assumptions.
  • Verifier adoption objective: evaluate signatures, freshness, endpoint state, and replication consistency using explicit algorithmic checks.
  • Host adoption objective: operate open replication origins that implement deterministic gossip and conflict handling behavior.
  • Governance objective: define mandatory minimum publication floors where needed, without converting those floors into exclusive host ceilings.

Terminology

governance area
A legal and administrative domain that issues and governs identity items and their verification rules.
physical identity item
A physical identity document or token issued under a governance area and recognized by that area for identity purposes.
chip-backed private key
Private-key material whose generation, storage, and signing operations are enforced by secure hardware boundaries.
governance verification endpoint
A network endpoint exposing discovery and/or status information needed to validate document legitimacy within a governance area. In this profile, those endpoints are exposed under /.well-known/gidas/.
MRZ
Machine Readable Zone data defined in ICAO Doc 9303 [ICAO-9303] [ICAO-9303-P3].
PID (person identification data)
A set of data that identifies a person under applicable law or standards contexts.
PID hash
A cryptographic hash over canonicalized PID fields extracted from MRZ-compatible document data.
MRZ hash
In GDIS v1, an alias for PID hash. MRZ is used to collect PID input, and the hash is computed once.
GDIS binding credential
A verifiable credential binding a PID hash to a subject DID, issuer evidence, and governance verification material.
GDIS subject DID
Subject DID in a GDIS binding credential, controlled by a GQSCD-class controller.
GQSCD controller
DID controller profile requiring protected key custody and signer-intent controls as defined by GQSCD Core [GQSCD-CORE].
decentralized event log
Open-participation append-only event replication layer used to publish and gossip GDIS verification material across many origins.
GDIS host
An origin participating in the decentralized event log by storing, serving, and gossiping signed events.
mandatory minimum hosts
Governance-required publication targets that define a minimum publication floor and not an exclusive publication ceiling.
publication set
Union of mandatory governance hosts and person-selected additional hosts to which a subject's materials are published.
evidence artifact
A signed or hashed machine-verifiable artifact used by issuers, hosts, or verifiers to substantiate a security or status claim.
conformance report
Implementation report that states conformance class coverage, policy assumptions, test outputs, and produced or consumed evidence artifact types.
verification result
Deterministic output object produced by a verifier after executing the normative processing rules in this specification.
web profile
Primary interoperability profile based on globally deployed web standards and internet protocols.
EU compatibility profile
Mapping layer from web profile outputs to EU-specific legal and ecosystem requirements.

Mandate (Law/Standards)

External Fact Layer

This section states legal/standards facts as external constraints. GDIS does not redefine these facts; it models and references them.

Physical Issuance and Governance Recognition

A GDIS root identity anchor MUST be a physical identity item issued under a governance area. Any claim that such an item is mandated, recognized, qualified, notified, or trusted MUST be backed by authoritative registries or legal texts for that governance area, such as EUR-Lex acts, trusted lists, or official notification dashboards [EU-910/2014] [EU-2024/1183] [EU-TRUSTED-LISTS] [EU-QSCD-NOTIFICATIONS].

EU Term Discipline (when EU context is used)

If implementers use EU eIDAS vocabulary (for example "electronic identification", "electronic identification means", "person identification data"), those usages SHOULD be anchored to Article 3 definitions and related amendments [EU-910-CONS] [EU-2024/1183].

MRZ Source Constraint

MRZ semantics are defined by ICAO Doc 9303 and MUST NOT be replaced by ad hoc field interpretations [ICAO-9303] [ICAO-9303-P3].

Legal Effect vs Technical Assurance

GDIS MAY claim technical comparability or stronger verifier cryptographic evidence, but it MUST NOT claim legal equivalence unless legal text explicitly supports that conclusion [EU-910/2014] [EU-910-CONS].

Design Proposal (GDIS Model)

Architecture Layer

This section is the technical design proposal for GDIS. It defines how physical identity anchors become portable cryptographic evidence.

Actors

  • Subject: person controlling a GQSCD-backed DID and deciding additional publication hosts.
  • Issuer: governance authority or delegated issuer producing signed GDIS binding credentials.
  • Verifier: relying party validating evidence, endpoint freshness, and event-log consistency.
  • Host: any open participant origin implementing GDIS host replication endpoints.

Lifecycle

  1. A physical item is inspected and chip/authenticity data is read under governance verification procedures.
  2. PID fields are extracted from MRZ-compatible data, canonicalized, and hashed into a single claim value.
  3. Governance verification endpoint data is normalized and hashed.
  4. The derived PID hash is bound to endpoint evidence for issuer and verifier checks.
  5. An issuer creates a GDIS binding credential addressed to a GDIS subject DID with GQSCD controller evidence.
  6. PID-hash key material and verification material are published to a publication set in the decentralized event log.
  7. Verifiers resolve events mechanically and evaluate binding validity, status, and freshness.

PID Hash and Endpoint Binding

GDIS v1 defines the binding input tuple as: (pidHash, endpointHash).

  1. Normalize MRZ text to ICAO-compatible uppercase machine-readable form and extract profile-defined PID fields from MRZ-compatible document data.
  2. Canonicalize PID JSON using JCS [RFC8785]; compute pidHash = SHA-256(pidCanonicalBytes). In v1, mrzHash is an optional alias and equals pidHash.
  3. Normalize endpoint URI per RFC 3986 [RFC3986]; compute endpointHash = SHA-256(endpointBytes).
  4. Compute bindingDigest = SHA-256("gdis:v1" || pidHash || endpointHash).
  5. Publish pidHash as the primary subject key and include bindingDigest as optional integrity metadata.

Credential Binding Model

The binding MUST be materialized as a W3C VC [VC-DM2] issued to a DID [DID-CORE] and signed using Data Integrity or JOSE-compatible proof suites [VC-DI] [RFC7515].

    "endpoint": "https://registry.example.gov/.well-known/gidas/verify",

GDIS event material is published as signed append-only events. Any origin MAY join as a GDIS host by implementing this profile's host API and verification rules.

Evidence Artifact Model

Conforming implementations MUST produce and process explicit evidence artifact objects. Artifact schemas MAY evolve by profile version, but required semantics and verification checks in this section are stable.

Endpoint status artifact endpoint URI, status payload, timestamp, signer data Governance endpoint operator Verifier MUST satisfy verifier freshness policy at decision time.

Normative Requirements

1. Physical Anchor Requirements

  • A GDIS issuance flow MUST begin from a physical identity item issued in a governance area.
  • The item MUST support secure key custody, including a chip-backed private key or equivalent protected key boundary.
  • The item MUST provide or resolve to governance verification endpoint data sufficient for legitimacy verification in that governance context.

2. MRZ-Derived PID Hash Requirements

  • Implementations MUST derive PID hash from PID fields extracted from MRZ data defined by ICAO Doc 9303 structures.
  • PID hash input fields MUST be explicitly versioned and canonicalized before hashing.
  • If an implementation emits MRZ hash, it MUST equal PID hash byte-for-byte.
  • Raw MRZ strings and raw PID fields MUST NOT be published in cleartext in the decentralized event log.

3. Verification Endpoint Requirements

  • GDIS HTTP endpoints MUST use the URI namespace /.well-known/gidas/<method | flow | operation>.
  • Endpoint URIs MUST use HTTPS and MUST be normalized before hashing.
  • Endpoint resources MUST provide at least discovery metadata and status/revocation data.
  • Implementations MUST NOT require non-/.well-known/gidas/ paths for normative interoperability behavior.
  • Verifiers MUST enforce freshness policy on endpoint status responses.

4. VC to DID Binding and Controller Requirements

  • The binding statement MUST be a GDIS binding credential issued to a GDIS subject DID.
  • The subject DID controller MUST satisfy a GQSCD controller profile [GQSCD-CORE].
  • Issuance evidence MUST include proofs for issuer authenticity, subject binding, status freshness, and replay resistance.

5. Event Log Replication and Openness

  • Replication participation MUST be open: any operator MAY run a GDIS host.
  • A governance profile MAY define mandatory minimum hosts.
  • Mandatory hosts MUST be interpreted as a minimum publication floor and MUST NOT be treated as an exclusive maximum set.
  • Subjects MUST be able to add person-selected hosts beyond mandatory minimum hosts.

6. Event Ordering, Integrity, and Rotation

  • Each event MUST include content digest, signature, PID hash, sequence number, and predecessor references.
  • Key rotation MUST be represented as a signed rotation event, ideally signed by prior authority key(s).
  • Revocation MUST be represented as explicit status events and reflected in verifier status output.

7. Mechanical Conflict Resolution Rules

Verifiers MUST execute the following deterministic process for each PID hash conflict set:

  1. Discard events with invalid signatures, unsupported schema versions, or invalid timestamps.
  2. Build branches from predecessor links and discard branches with broken ancestry.
  3. Determine authoritative issuer key state from latest valid key-rotation lineage.
  4. For same semantic field and sequence value, choose the candidate with latest createdAt; if tied, choose the lexicographically smaller eventId.
  5. Any valid revocation event at or after effective binding sequence sets status to revoked.

8. Minimum Host API

Conforming hosts MUST expose:

  • GET /.well-known/gidas/events/{pidHash}
  • GET /.well-known/gidas/event/{eventId}
  • POST /.well-known/gidas/gossip
  • GET /.well-known/gidas/status/{pidHash}

9. Replay and Freshness Controls

  • Verification payloads MUST include nonce or challenge binding and verifier audience binding.
  • Replay windows MUST be bounded by policy and checked at verification time.
  • Verifiers MUST refuse proofs outside accepted timestamp skew and freshness windows.

Processing Model

Issuer Processing Rules

  1. The issuer MUST verify the physical anchor evidence, including governance issuance context, before producing any GDIS binding credential.
  2. The issuer MUST compute PID hash and endpoint-binding values exactly as defined in PID Hash and Endpoint Binding.
  3. The issuer MUST include enough evidence references for a verifier to check issuer authenticity, status freshness, and replay resistance.
  4. The issuer MUST sign issuance artifacts with currently authoritative keys and publish events compatible with host acceptance rules.

Host Processing Rules

  1. A host MUST validate event schema version, signature, sequence, and predecessor references before persisting an event.
  2. A host MUST reject events with invalid signatures, broken ancestry, or unsupported version markers.
  3. A host MUST expose deterministic event retrieval results for the same local view and request parameters.
  4. A host SHOULD persist rejection reasons for operational audit and interoperability debugging.

Verifier Processing Rules and Output Contract

A verifier MUST execute requirements from sections 3, 5, 6, 7, and 9 before returning a decision. The verifier output MUST be explicit and machine-consumable. 

{
  "pidHash": "sha256:...",
  "decision": "valid",
  "status": "active",
  "evaluatedAt": "2026-02-17T12:34:56Z",
  "issuerKeyState": "trusted",
  "freshness": {
    "endpointOk": true,
    "withinPolicyWindow": true
  },
  "eventResolution": {
    "selectedHead": "sha256:...",
    "branchCount": 2
  },
  "replayCheck": {
    "nonceBound": true,
    "audienceBound": true
  }
}

If decision is not valid, verifiers MUST return an explicit failure reason code and MUST NOT collapse all failures into one opaque error.

Conformance Reporting Requirements

Implementations claiming conformance MUST publish or provide a conformance report containing at least:

  • declared conformance class (Issuer, Host, Verifier, or multiple);
  • supported profile and schema versions;
  • implemented requirement units and any documented exclusions;
  • test evidence, including pass/fail results and artifact samples;
  • security assumptions, clock/freshness policy values, and trust anchor policy inputs.

Reports SHOULD include machine-readable fields so independent tooling can compare behavior across implementations.

Web Profile (Mandatory) and Compatibility Layers

Primary Web Profile

Implementations MUST treat web standards as the primary interoperability baseline:

  • DID Core [DID-CORE];
  • VC Data Model 2.0 [VC-DM2];
  • VC Data Integrity or JOSE/COSE proof containers [VC-DI] [RFC7515] [RFC9052];
  • HTTP transport semantics [RFC9110].

Threat Model Honesty

Implementations MUST assume hostile client/network environments unless security boundaries are cryptographically enforced.

At minimum, conforming designs MUST model key exfiltration attempts, endpoint spoofing, replay attempts, event-log inconsistency, and policy abuse where governance minimum host requirements are misused as maximum exclusivity constraints.

Threat Required Control Evidence Type
Key exfiltration Hardware-backed non-exportability + controller attestation Device or controller evidence chain
Endpoint spoofing Endpoint hash binding + TLS + issuer signatures Bound endpoint hash and signed metadata
Replay of proof artifacts Nonce/audience/timestamp checks Verifier transcript and challenge log
Registry manipulation Multi-host reconciliation + deterministic conflict rules Host receipts and reconciled event view
Governance over-closure of host set Mandatory host sets treated only as publication floor Publication policy and host-distribution record

"Approved software lists" or governance labels alone MUST NOT be treated as technical guarantees without cryptographic enforcement.

Conformance reports SHOULD list which threats were tested directly and which threats are mitigated by inherited platform controls.

Privacy Considerations

  • Event logs SHOULD store hashes, references, and proof digests instead of raw identity fields.
  • Raw MRZ/PID material MUST remain off-log unless law explicitly requires disclosure.
  • Selective disclosure mechanisms SHOULD be preferred for verifier transactions.
  • Implementations SHOULD support pairwise verifier challenges to reduce cross-context linkability.

Open Issues

Define a compact canonical PID field set for v1 that is stable across passport and national ID form factors while keeping jurisdiction mapping explicit.

Define a portable host gossip anti-spam model that preserves open host admission and does not require centralized host approval.

Define optional pairwise identifier blinding mode that preserves global deduplication properties needed by governance profiles.

Decide whether v1 transport profile includes only HTTP(S) or also content-addressed publication channels (for example IPFS) as first-class options.
